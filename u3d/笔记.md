# UI


## Button


### 🧩 Button 组件核心构成

一个标准 Button 是以下三部分的整合：

1. **视觉表现 (Image)**：显示按钮的背景图片，是射线检测的默认目标。
2. **状态反馈 (Transition)**：根据交互状态（普通、高亮、按下、禁用、选中）切换视觉样式。常用 **Color Tint**（颜色变化）、**Sprite Swap**（切换图片）或 **Animation**（播放动画）模式。
3. **输入响应 (Event)**：通过 `OnClick()` 事件，将用户的点击操作与脚本逻辑连接起来。

### 🎬 四种过渡模式 (Transition) 详解

`Transition` 属性定义了按钮如何响应用户交互，理解它有助于提升用户体验。


| Transition 模式 | 作用                                    | 典型参数                  | 适用场景                                   |
| --------------- | --------------------------------------- | ------------------------- | ------------------------------------------ |
| **None**        | 无视觉变化，只执行点击逻辑。            | -                         | 纯逻辑触发按钮，通常用法或代码控制视觉。   |
| **Color Tint**  | 通过改变目标Graphic的颜色来提供反馈。   | 不同状态的颜色、渐变时间  | **最常用**，资源开销小，适合大多数标准UI。 |
| **Sprite Swap** | 根据状态切换为指定的精灵图片。          | 不同状态的Sprite          | 需要更丰富视觉反馈（如立体按钮按下）时。   |
| **Animation**   | 播放Unity动画系统(Animator)制作的动画。 | 关联的Animator Controller | 实现复杂的动态效果（如按钮震动、呼吸灯）。 |

### 👆 按钮的三种状态 (Interactable)

`Interactable` 属性控制按钮的活性。

* **启用 (true)**：按钮正常工作，能接收点击事件，并根据 `Transition` 表现视觉反馈。
* **禁用 (false)**：按钮变为不可交互状态，不再响应点击，并立即进入 `Disabled` 状态下的视觉样式（通常是灰色）。在冷却、条件未达成时常用。

#### 🧭 键盘/手柄导航 (Navigation)

`Navigation` 属性定义了玩家使用键盘方向键或手柄摇杆时，焦点在按钮间的移动逻辑。

* **None**：不自动导航，需鼠标/代码控制。
* **Horizontal / Vertical**：限制在水平或垂直方向导航。
* **Automatic**：Unity自动推断最佳导航目标。
* **Explicit**：为每个方向手动指定目标按钮。

### ⚡️ 如何绑定点击事件 (三种方式)

想要按钮点击后执行代码，你需要建立“点击”和“逻辑”之间的桥梁。


| 绑定方式                  | 操作方法                                                                                                                      | 优点                                                   | 适用场景                                                               |
| ------------------------- | ----------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------ | ---------------------------------------------------------------------- |
| **1. Inspector 拖拽绑定** | 1. 在`OnClick()`事件列表下方点“+”。<br/>2. 将挂载了目标脚本的物体拖入对象框。<br/>3. 从下拉菜单中选择要调用的`public`方法。 | 可视化操作，无需编写代码，适合设计师和逻辑简单的项目。 | 场景中固定的UI按钮，如“开始游戏”、“设置”。                         |
| **2. 代码动态绑定**       | 在脚本中获取Button组件，调用`button.onClick.AddListener(方法名)`。                                                            | 灵活，可在运行时动态增减、修改事件。                   | 运行时动态生成的按钮，如背包格子、商店列表项。                         |
| **3. 独立脚本组件**       | 为按钮创建专属脚本，在`Awake`或`Start`中获取自身Button并绑定。                                                                | 高内聚，遵循单一职责原则，便于维护和复用。             | 具有复杂独立逻辑的按钮，如“登录”（含网络请求）、“提交”（含冷却）。 |

### 💡 进阶应用场景

* **创建不规则形状按钮**：默认按钮的点击区域为矩形。利用 `Image.alphaHitTestMinimumThreshold` 属性，可以让按钮只在图片的非透明像素区域响应点击。启用该属性需要将关联图片纹理的`Read/Write Enabled`选项开启。
* **实现按钮长按检测**：通过实现 `IPointerDownHandler` 和 `IPointerUpHandler` 接口，在`OnPointerDown`时启动一个计时器，当 `OnPointerUp` 触发且持续时间达到阈值时，判定为长按。
* **制作按钮组 (Toggle Group)**：实现类似单选框的效果，确保一组按钮中只能有一个被选中。通常与 `Toggle` 组件搭配实现。

### 🛑 常见坑点与排查 (点击无反应/状态异常)

#### 原因和解决方法速查表


| 原因                   | 检查项与解决方案                                                                                                                                                                                                           |
| ---------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **环境与组件缺失**     | 场景中需要有**EventSystem**，UI元素所在的Canvas组件上需要有**Graphic Raycaster**脚本。                                                                                                                                     |
| **Button视觉组件缺失** | Button 本身或其子物体上需要有一个**Graphic**组件（如Image或Text），并确保其 `Raycast Target` 属性已勾选，否则射线无法命中。                                                                                                |
| **UI遮挡与层级问题**   | - 检查是否有其他UI元素（如半透明面板）在按钮**上方**并遮挡了它。可临时调整它们在Hierarchy中的顺序排查。<br/>- 检查Button的父物体是否被禁用或被设置了 **Canvas Group** 并勾选了 `Ignore Parent Groups`。                    |
| **空间变换异常**       | - 确保Button没有被旋转到**背面**（Y轴为180度）。可以在`Graphic Raycaster`组件中取消 `Ignore Reversed Graphics` 选项临时解决。<br/>- 确保坐标轴的 `Scale` 值**不为负数**，负的Scale也会导致射线检测异常。                   |
| **属性与代码错误**     | - 检查`Interactable` 是否被误关闭。<br/>- 检查 `OnClick()` 事件列表中绑定的方法是否因脚本编译错误、方法签名变更或引用的GameObject被删除而变为`Missing`。<br/>- 检查是否在代码中多次 `AddListener` 导致相同方法被重复调用。 |

### 🚀 性能优化指南

* **减少Raycast Target**：对于只需显示的图片和文本，**务必关闭其 `Raycast Target`** 属性，以减少射线检测的开销。
* **优化渲染批次**：
  * **合批(Batching)**：尽量让相邻的UI元素使用**相同的材质和纹理图集**，以减少绘制调用。
  * **使用RectMask2D**：在滚动视图（ScrollView）中，使用`RectMask2D`代替`Mask`组件，后者会强制UI元素持续渲染并消耗更多GPU资源。
* **限制动画组件**：避免为UI中的按钮使用`Animator`组件做简单动画，这会带来不必要的性能开销。推荐使用轻量级动画方案，如 DoTween 插件。
* **注意UI数量**：**避免在同一个Canvas下动态创建和销毁大量按钮**，这会引起频繁的网格重建，消耗大量CPU。
* **禁用不可见的按钮**：对于因翻页或滚动而离开屏幕的按钮组，可以通过代码设置其父物体的`GameObject`为`SetActive(false)`，或者利用对象池循环使用，以彻底禁用其逻辑。


## Canvas

`Canvas` 是 Unity UI 系统的核心容器。所有 UI 元素（如 `Image`, `Text`, `Button`）都必须是 Canvas 的子物体，并由 Canvas 负责渲染和交互射线检测。

---

### 📌 Canvas 的三种渲染模式


| 模式                       | 渲染空间                                             | 使用场景                                                               | 关键点                                                       |
| -------------------------- | ---------------------------------------------------- | ---------------------------------------------------------------------- | ------------------------------------------------------------ |
| **Screen Space - Overlay** | 屏幕空间，始终在最前面，独立于场景摄像机。           | 主菜单、HUD、血条、对话框等所有**2D界面**。                            | 无需指定摄像机，自动覆盖全屏，被场景物体穿透但无法穿透回来。 |
| **Screen Space - Camera**  | 屏幕空间，但受指定摄像机视野影响，可被场景物体遮挡。 | 需要与 3D 场景融合的 UI（例如准星受光照影响、3D 角色旁边的对话气泡）。 | 须指定`Render Camera`；UI 会随摄像机 FOV 和距离有透视效果。  |
| **World Space**            | 世界空间中的平面，如同一张画板悬浮在 3D 世界中。     | 3D 世界里的交互屏幕（电脑显示屏、NPC 头顶名字、可交互的指示牌）。      | UI 可自由旋转、缩放，会受场景光照和遮挡影响。                |

> **如何选择？** 绝大多数情况用 `Overlay`；需要 UI 被 3D 物体遮挡或用摄像机透视效果时用 `Screen Space - Camera`；需要把 UI 放进 3D 场景中作为物体一部分时用 `World Space`。

---

### 🧩 Canvas 的子组件与关键依赖


| 组件                  | 作用                                              | 备注                                                                                              |
| --------------------- | ------------------------------------------------- | ------------------------------------------------------------------------------------------------- |
| **Canvas Scaler**     | 控制 UI 在不同分辨率下的缩放适应方式。            | **必配组件**，推荐 `UI Scale Mode = Scale With Screen Size`，`Reference Resolution = 1920x1080`。 |
| **Graphic Raycaster** | 检测鼠标/触摸在 UI 上的点击，实现 Button 等交互。 | 默认自带，不要删除。                                                                              |
| **Rect Transform**    | Canvas 自身的变换组件，但受渲染模式约束。         | 在 Overlay 模式下，Rect Transform 的位置/大小无效（始终填满屏幕）。                               |

#### Canvas Scaler 关键设置


| 参数                   | 推荐值                     | 说明                                            |
| ---------------------- | -------------------------- | ----------------------------------------------- |
| `UI Scale Mode`        | **Scale With Screen Size** | 根据屏幕尺寸等比例缩放 UI，保持设计时比例。     |
| `Reference Resolution` | `1920 x 1080`              | 以此为基准设计 UI，其它分辨率等比例缩放。       |
| `Screen Match Mode`    | `Match Width or Height`    | 常用，0 匹配宽度，1 匹配高度，一般设 0.5\~0.7。 |
| `Match`                | `0.5`                      | 平衡宽高变化，避免 UI 变形。                    |

---

### 🎨 核心概念：Rect Transform

所有 UI 元素都有 `Rect Transform`，它比普通 `Transform` 多了锚点(Anchors)、轴心(Pivot)和大小调整。


| 属性                | 作用                                                                               |
| ------------------- | ---------------------------------------------------------------------------------- |
| **Anchors（锚点）** | 定义 UI 元素相对于父级四个角的固定点。例如锚点全在左上角，则元素相对于左上角定位。 |
| **Pivot（轴心）**   | UI 元素自身的旋转/缩放中心点，也是坐标原点。                                       |
| **Pos X / Pos Y**   | 元素锚点与自身轴心的偏移量。                                                       |
| **Width / Height**  | 元素宽高。                                                                         |

**常用锚点预设**：在 Rect Transform 左上角可快速选择“居中”、“拉伸填满”等。

---

### 🧰 典型应用场景

1. **主菜单界面 (Main Menu)**
   * 模式：`Screen Space - Overlay`
   * 分辨率适配：`Canvas Scaler` 保证各个屏幕比例下按钮位置正确。
   * 常见组件：`Button`, `Image`, `Text`, `Scroll View`。
2. **游戏内 HUD（血条、技能图标）**
   * 模式：`Overlay`
   * 优化：血条用 `Slider` + `Image.FillMethod`，技能冷却使用 `Image` 的 `Fill Amount`，避免每帧重建。
   * 跟随角色：使用世界坐标转屏幕坐标的方式，将 3D 角色头顶血条放在独立 Overlay Canvas 上（需要每帧更新位置）。另一种方式是用 World Space Canvas 直接挂在角色身上。
3. **3D 世界中的交互屏幕（World Space Canvas）**
   * 模式：`World Space`
   * 应用：电脑显示器、商店面板、可点击的公告牌。
   * 注意事项：需调整 `Scale` 和 `Transform` 使其与物体对齐；交互射线会与 3D 物理射线冲突，需设置合适的 `Event Camera`。
4. **动态 UI（背包、对话）**
   * 通过代码动态实例化 `Button`、`Image` 等，注意及时回收内存。
   * 大量动态生成时使用 **对象池** 并注意 Canvas 的批量重建开销。

### ⚠️ 常见误区与坑


| 误区                                                   | 后果                                                             | 正确做法                                                                                                                                  |
| ------------------------------------------------------ | ---------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------- |
| **过度使用嵌套 Canvas**                                | 每个子 Canvas 都会独立合批，增加 DrawCall。                      | 尽量使用单个 Canvas，内部通过 UI 组件层级控制渲染顺序。仅在需要不同渲染模式或独立射线区域时才用子 Canvas。                                |
| **频繁修改 UI 元素的 Position / Scale / Color 等属性** | 触发整个 Canvas 的重建（Rebuild），开销极大。                    | 对于需要持续变化的 UI（如血条、倒计时），使用`Slider`、`Image.fillAmount`、`Text.text` 不会触发布局重建；但是改变位置、大小、父级会触发。 |
| **Canvas 上挂载大量 UI 元素（数千个）**                | 每次 UI 变化都会遍历所有元素，造成卡顿。                         | 使用分页、对象池、动态加载；对不活跃的 UI 设置`GameObject.SetActive(false)`。                                                             |
| **在同一 Canvas 中混合大量半透明重叠图像**             | GPU 渲染负担增加（Overdraw），尤其移动端。                       | 尽量减少完全重叠的透明元素；能用纯色就用纯色；合理使用`RectMask2D` 剔除不可见区域。                                                       |
| **忽略 Raycast Target 设置**                           | 不可交互的图像也参与射线检测，浪费性能，且可能拦截下层 UI 点击。 | 对于`Image`、`Text` 等不参与点击的组件，关闭 `Raycast Target`。                                                                           |
| **世界空间 Canvas 未调整缩放**                         | 字太小或太大，难以点击。                                         | 调整 Canvas 的`Scale` 和 `Rect Transform` 大小，通常需要设置一个合理的 `Dynamic Pixels Per Unit`（如 10）。                               |
| **Canvas Scaler 设置错误导致 UI 模糊或超出屏幕**       | 在非 16:9 分辨率下 UI 错位、显示不全。                           | 正确设置 Reference Resolution，并选择合适的 Match 值。                                                                                    |

### 🚀 性能优化技巧

#### 1. 动静分离（关键优化）

将**不常变化**的 UI 元素（背景、静态图标）放在一个 Canvas，**经常变化**的 UI（血条、倒计时、动态数值）放在另一个独立的 Canvas。这样每个 Canvas 的网格重建互不影响，大大降低开销。

**示例结构：**

**text**

```
- Canvas_Static (Overlay)
   - Background
   - Frame
   - Button (可交互，但按钮文字不常变)
- Canvas_Dynamic (Overlay)
   - HealthBar (Slider)
   - ScoreText (Text)
```

#### 2. 使用 UI 组件内置功能代替手动刷新

* 血条 → `Slider` + 不勾选 `Whole Numbers`
* 技能冷却 → `Image.fillAmount`
* 简单的动画 → `Animator` 或 `UIEffect` 插件，避免代码每帧修改属性

#### 3. 减少 Layout Group（Horizontal/Vertical/Grid Layout Group）

Layout Group 会每帧强制计算子物体位置，开销较大。如果布局固定，手动摆放；如果必须用，确保子物体数量少，且尽量不要动态增减。

#### 4. 禁用不可见的 UI

* 关闭整个页面的 `GameObject` 而非仅置透明。
* 使用 `CanvasGroup.alpha` 并配合 `blocksRaycasts = false` 可防止射线，但不会节省重建开销。

#### 5. 图集（Sprite Atlas）

将多个小图片打包成图集，可以减少 DrawCall 和内存。Unity 会自动合并同图集的图片，但注意 Canvas 内的图片如果来自不同图集且混合渲染，会增加 DrawCall。

#### 6. 使用 RectMask2D 代替 Mask

* `Mask` 组件会创建新的渲染队列，打断合批。
* `RectMask2D` 性能更好，适合裁剪滚动视图。

#### 7. 降低 Canvas 的更新频率

对于只需要每秒刷新几次的 UI（如时间显示），可以手动控制刷新间隔，不要每帧更新 `Text`。

**csharp**

```
float timer = 0f;
void Update() {
    timer += Time.deltaTime;
    if (timer >= 0.5f) {
        timer = 0f;
        UpdateUIText();
    }
}
```

### 🔧 调试工具与技巧

* **Frame Debugger**（Window → Analysis → Frame Debugger）
  可看到 Canvas 的合批（Batch）情况，检查是否因为打断合批导致 DrawCall 升高。
* **UI Profiler**（Profiler 窗口 → UI 模块）
  查看 Canvas 重建耗时、布局计算耗时等。
* **统计面板（Stats）**
  在 Game 视图中点击 Stats 按钮，可看到 Batches（批次）和 Tris（三角面），如果 Batches 过高就要优化。

### 📚 总结


| 重点               | 要点                                                                  |
| ------------------ | --------------------------------------------------------------------- |
| **Canvas 模式**    | 多数用 Overlay；需与 3D 融合用 Screen Space - Camera 或 World Space。 |
| **Canvas Scaler**  | 保证分辨率适配，必配。                                                |
| **Rect Transform** | 理解锚点和轴心，才能精确控制 UI 位置。                                |
| **性能核心**       | 动静分离 Canvas，减少属性变化，关闭无用 Raycast Target。              |
| **常见坑**         | 频繁改变位置/大小/父级触发重建；忽略 Canvas 嵌套导致的性能问题。      |

掌握 Canvas 是开发 Unity UI 的基础。从简单 HUD 到复杂的 RPG 背包系统，都可以通过合理的 Canvas 结构和优化技巧实现流畅的交互界面。


# 组件

组件可以理解成一个对象的功能。比如mesh、材质、刚体、灯光等等。对象之间的区别就在于挂载的组件不一样。特殊的一点是所有的对象都有transform(位置组件)，因为这是物体存在于空间的标识。

## 拖尾渲染器


Trail Renderer 是 Unity 自带的一个轻量级 3D 拖尾组件，专门用来为移动物体生成动态、连续的轨迹效果。它的核心原理是在物体移动时，按一定时间或距离间隔采样其位置，将这些历史点连接成一条连续的网格面片进行渲染。

你可以通过 `GameObject > Effects > Trail` 创建它，然后将其拖拽为需要产生拖尾的物体的子对象即可。在调试时，不运行游戏直接在 Scene 视图中移动物体，即可预览拖尾，非常方便。

### 📑 Trail Renderer 核心参数速查


| 参数                    | 功能                                                                         | 优化建议/备注                                                              |
| ----------------------- | ---------------------------------------------------------------------------- | -------------------------------------------------------------------------- |
| **Time**                | 拖尾段的生命周期（秒），即拖尾的总长度。                                     | 性能关键，较长的拖尾需要更多的顶点来维持平滑度。                           |
| **Width**               | 拖尾的宽度，可设置曲线，使尾部渐细。                                         | 视觉关键，调整曲线可创建自然渐隐效果。                                     |
| **Color**               | 拖尾的颜色，可设置渐变，实现边缘淡出。                                       | 视觉关键，常用于制作从实色到透明的效果。                                   |
| **Min Vertex Distance** | 生成新顶点的最小位移距离。数值越小，拖尾越平滑，但顶点数越多，性能开销越大。 | **首要性能优化点**。在保证效果的前提下，使用尽可能大的值。                 |
| **Alignment**           | 拖尾的朝向模式，可设置为`View`（始终面向摄像机）或 `TransformZ`。            | 3D 拖尾常用`View` 模式；2D 拖尾如需特殊效果可调整。                        |
| **Texture Mode**        | 纹理映射模式，如`Stretch`（拉伸）、`Tile`（平铺）。                          | 视觉关键，`Tile` 模式配合流动纹理可制作火柱等特殊效果。                    |
| **Corner Vertices**     | 路径拐角的额外顶点数，数值越高，拐角越圆滑。                                 | 视觉优化，值越高性能消耗越大。                                             |
| **End Cap Vertices**    | 拖尾末端的额外顶点数，用于使末端更圆润。                                     | 视觉优化，值越高性能消耗越大。                                             |
| **Emitting**            | 实时开关拖尾的生成，可暂停或恢复拖尾效果。                                   | **运行时性能关键点**。在物体静止或无需拖尾时，务必将其 `false`。           |
| **Materials**           | 拖尾渲染所用的材质。                                                         | 材质 Shader 复杂程度直接影响 GPU 性能。                                    |
| **`Clear()` 方法**      | 立即清除拖尾的所有现有线段。                                                 | **运行时强制清除缓存**。例如，在物体瞬间移动后调用，避免产生“鬼影”拖尾。 |

### 🧰 典型应用场景

Trail Renderer 的应用非常广泛，几乎所有强调“运动感”的效果都能用它实现。以下是一些常见应用场景：


| 场景                  | 玩法/作用                                    | 实现关键点                                                                                   |
| --------------------- | -------------------------------------------- | -------------------------------------------------------------------------------------------- |
| **武器刀光/剑气**     | 突出挥舞轨迹，增强打击感。                   | **同步动画**：通过脚本在特定动画帧开启/关闭 `Emitting` 属性。                                |
| **子弹/导弹轨迹**     | 标示投射物路径，方便预判。                   | **生命周期管理**：按需启用，子弹命中时先 `Clear()` 再禁用，避免重复利用时拖尾异常。          |
| **角色冲刺/瞬移残影** | 凸显高速移动，打造帅气技能特效。             | **短暂开启**：在冲刺瞬间短暂启用 `Emitting`，角色闪烁或冲刺动作结束后关闭。                  |
| **赛车尾焰/轮胎烟雾** | 丰富载具细节，提升竞速游戏速度感。           | **固定材质**：通常使用带有火焰纹理的材质，并配合 `Color` 渐变和 `Width` 曲线，使其自然衰减。 |
| **物体运动辅助线**    | 高亮路径，展示引导线，常用于解谜或动作游戏。 | **长拖尾设定**：将 `Time` 值设为较大值，并确保拖尾平滑且可见。                               |
| **魔法光效**          | 释放法术时留下绚丽轨迹，增强特效表现力。     | **与 Shader 配合**：使用特制的粒子 Shader 材质，并配合 `Color` 渐变动画。                    |

### 🚧 常见误区与避坑指南


| 问题现象                   | 原因排查                                                                              | 解决方案                                                                                           |
| -------------------------- | ------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------- |
| **拖尾不显示**             | 材质未配置，或材质 Shader 不支持拖尾渲染。                                            | 为 Trail Renderer 指定材质，并使用内置的**标准粒子着色器**，或 `Sprites/Default`（常用于2D游戏）。 |
| **拖尾显示为黑色方块**     | 材质 Shader 期望光照信息（如 Standard Shader）但场景无有效光源。                      | 将材质的 Shader 改为**`Sprites/Default`** 或 **`Unlit/Texture`**。                                 |
| **拖尾被其他物体遮挡**     | 3D 拖尾的渲染队列与 2D 精灵的排序规则冲突。                                           | 将材质的 Shader 改为 **`Sprites/Default`**，它在 2D 系统中排序正确。                               |
| **拖尾边缘有锯齿**         | `Min Vertex Distance` 值过大导致拖尾分段明显。                                        | 适当**减小`Min Vertex Distance`**，让采样更密集。                                                  |
| **拖尾断裂或不连续**       | `Min Vertex Distance` 过小会产生大量顶点导致性能问题；或拖尾末端 (`End Cap`) 太尖锐。 | 设置合适的`End Cap Vertices` 值。                                                                  |
| **物体速度极快时拖尾消失** | 速度太快，移动距离超过了`Min Vertex Distance`。                                       | 适当**减小 `Min Vertex Distance`** 值，或考虑使用射线扫描等更复杂的方法。                          |
| **拖尾“鬼影”残留**       | 物体瞬间移动（如归位、传送）后，旧拖尾会保留。                                        | 在移动后调用**`TrailRenderer.Clear()`** API 立即清除旧线段。                                       |

> **💡 提效小技巧**：如果你发现自己的拖尾设置适合多个地方使用，可以考虑将这些配置保存为 **预制体（Prefab）**，方便重复利用。如果觉得内置的拖尾效果不够灵活，也可以考虑使用粒子系统（Particle System）来模拟拖尾，虽然它的性能开销会高一些，但能实现更复杂、绚丽的动态效果。

### 🚀 性能优化技巧

Trail Renderer 的性能瓶颈在于**顶点数量**。优化核心在于「减少顶点数」与「降低每帧消耗」。


| 优化类别               | 核心优化方法                                                                                                                        |
| ---------------------- | ----------------------------------------------------------------------------------------------------------------------------------- |
| **顶点数优化**         | **增大`Min Vertex Distance`**：在保证视觉平滑的前提下，使用尽可能大的值，这是最有效的优化手段。                                     |
| **生命周期管理**       | **使用 `Emitting` 开关**：在物体静止或无需拖尾时，务必通过代码将其设置为 `false`。                                                  |
| **材质与 Shader 优化** | **避免使用复杂 Shader**：尽量使用 `Unlit` 或 `Sprites/Default` 等简单的着色器，避免标准粒子材质。                                   |
| **更新频率优化**       | **减少位置更新频率**：对于无需每帧更新的物体（如路径点固定的机关），可降低其位置更新频率。                                          |
| **最佳实践**           | **新建空物体并只保留 Trail Renderer**：官方建议的实践，将拖尾作为一个独立的 GameObject 子物体，可以避免与父物体其他渲染器交互干扰。 |

### 🔄 Trail Renderer vs. Line Renderer 对比

两者算法相同，但设计目标完全不同。


| 维度         | Trail Renderer                                                       | Line Renderer                                  |
| ------------ | -------------------------------------------------------------------- | ---------------------------------------------- |
| **核心概念** | 动态生成的轨迹，记录物体运动的历史路径。                             | 手动定义的连续线段，由固定的顶点数组决定。     |
| **数据来源** | 组件内部采样。**开发者通过移动物体间接控制，无法直接设置顶点位置**。 | 开发者通过代码或动画直接控制顶点的数量与位置。 |
| **生命周期** | 自动管理，每个段点有`Time` 参数控制其消失时间。                      | 始终存在，除非开发者手动改变或禁用。           |
| **典型应用** | 自动跟随的刀光、子弹轨迹等。                                         | 静态的瞄准线、路径点连线、地形雷达扫描线等。   |
| **内存占用** | 运行时动态分配，越长的拖尾占用越高，**峰值可控**。                   | 在设计时决定，内存占用**固定且可预测**。       |

---

### 💎 总结

总的来说，想要用好 Trail Renderer，需要理解它“动态生成顶点”的本质。核心就是监控它的顶点数量，并通过 `Min Vertex Distance`、`Emitting`、`Time` 等参数做好精细化控制，以防其因顶点爆炸而拖垮性能。

`TrailRenderer` 是新人和高手都经常使用的组件，理解它的核心逻辑和优化思路后，你就能高效地实现想要的运动特效了。


## 线段渲染器


### 📌 核心参数与API速查

理解它的几个关键属性是正确使用它的基础。


| 参数                     | 说明                                                                                                                                            |
| ------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------- |
| `Positions`              | 线条的顶点数组。`Vertex Count`决定顶点数量，可以编辑每个顶点的具体位置。勾选`Use World Space`时，顶点位置基于世界坐标，否则基于物体的局部坐标。 |
| `Width`                  | 线条宽度，世界单位。可以用曲线控制宽度变化，例如设置从窄到宽的渐变。                                                                            |
| `Color`                  | 线条颜色，可以用渐变（`Color Gradient`）控制颜色变化。                                                                                          |
| `Material`               | 线条材质。用`Sprites/Default`材质可制作最简单的线条；使用`Particles/Standard Unlit`配合纹理，能实现虚线或流光效果。                             |
| `Loop`                   | 是否闭合线条。勾选后，线条会从最后一个顶点连接到第一个顶点，形成一个闭合图形。                                                                  |
| `Alignment`              | 对齐方式，推荐使用`View`（始终面向摄像机），这对激光、拖尾等效果至关重要。                                                                      |
| `Corner Vertices`        | 拐角圆滑度。增加该值，可以让线条的折角变得更圆润。                                                                                              |
| `Generate Lighting Data` | 是否生成法线以接受光照。关掉可节省性能。                                                                                                        |

**csharp**

```
// 创建并初始化LineRenderer
LineRenderer lr = gameObject.AddComponent<LineRenderer>();
lr.material = new Material(Shader.Find("Sprites/Default"));
lr.startWidth = 0.2f;
lr.endWidth = 0.2f;
lr.positionCount = 2;
lr.SetPosition(0, new Vector3(0, 0, 0));
lr.SetPosition(1, new Vector3(1, 0, 0));
```

### 🧰 应用场景

* **路径引导**：最常见的是为角色绘制导航路线或在场景中画出寻路轨迹，也可以编辑顶点来制作自动寻路的可视化效果。
* **技能/武器特效**：游戏里的激光、光束、魔法链等，都可以通过`LineRenderer`快速实现。
* **工具与调试**：在编辑器中常用来绘制辅助线或边界，运行时动态生成攻击范围、碰撞预警线等，帮助调试。
* **UI/界面元素**：用于绘制UI框、进度条边框等，比纯UI方案更节省性能。
* **创意/模拟效果**：例如模拟电线下垂、弹弓/弓箭的轨迹线等。

### 🚧 常见误区与避坑指南

* **❌ 线条不显示？**
  * **检查1：顶点数量**：`Position Count`至少要大于等于2，线条才会被渲染。
  * **检查2：材质与Shader**：标准Shader（Standard）可能不支持线条渲染，最简单的解决办法是把材质的Shader换成`Sprites/Default`。
  * **检查3：摄像机剔除**：确认线条所在的Layer没有被摄像机的`Culling Mask`排除掉。
* **❌ 顶点过多，性能“吃不消”**：每帧动态调整大量顶点坐标，是性能问题的常见原因。解决思路是`更新频率控制`和`延迟LOD（Level of Detail，细节层级）`。
* **❌ 线条显示“一节一节的”，不够平滑**：需要在曲线的`细分`和`顶点圆角化`上下功夫。
  * 用**贝塞尔曲线**对生硬的折线进行平滑处理。
  * 或适当提高`Corner Vertices`参数值，让线条的转角变得更圆滑。
* **❌ 线条从摄像机视角看会消失或扭曲**：如果`Alignment`没设置为`View`，线条的边缘在某些角度下会消失；并且，线条的渲染顺序（RenderQueue）也应当高于遮挡物。

### 🛠️ 性能优化技巧

* **顶点数量控制**：顶点数是影响性能的主要因素。在保证视觉效果的前提下，应尽量减少顶点数，这是最直接的优化。
* **动态线条更新频率**：在游戏逻辑中，可以减少路径重计算的频率（如`0.2f`秒一次）；`Line Renderer`的顶点位置也无需每帧刷新，可设置一个最小距离阈值，当移动超过阈值时再更新。
* **材质与着色器**：尽量使用Unity内置的`Sprites/Default`、`Particles/Standard Unlit`等简单着色器，避免复杂的材质计算。
* **合并线条**：尽量复用`Line Renderer`对象，而非频繁创建销毁，以降低资源开销。
* **空间换时间**：在性能和效果上做好权衡。比如，在简单的直线路径上无需过度细分，但复杂的曲线则必须用更多顶点来保证效果。

### 📊 替代方案对比

有时`Line Renderer`不是最佳选择，可以根据具体需求考虑其他方案：


| 替代方案                | 特点                                                     | 适用场景                                         |
| ----------------------- | -------------------------------------------------------- | ------------------------------------------------ |
| **GL 或 Graphics.Draw** | 底层绘制，性能极佳，适合绘制大量一次性线条。             | 调试绘制、辅助线、不经常变化的线条。             |
| **Mesh + MeshRenderer** | 最灵活可控，可实现自定义管线、大幅降低DrawCall。         | 复杂的线框模型、大规模线条渲染、极致的性能优化。 |
| **Trail Renderer**      | 动态生成拖尾效果。                                       | 高速运动的拖尾、刀光、粒子动态轨迹。             |
| **Shader Graph 方案**   | 可视化的着色器制作，可以做出非常炫酷的发光、溶解等效果。 | 对线条视觉效果有特殊定制需求时。                 |

### 💎 总结与建议

总的来说，`Line Renderer`用好了是利器，用不好则可能成为性能瓶颈。

* **简单路径/少量线条**：直接使用 `LineRenderer` 是最佳选择。
* **动态/变化多的线条**：首要考虑**顶点数量**和**更新频率**的优化。
* **数量多或性能要求极高**：可评估 `Mesh` 方案或其他替代品。

## 物理材质


### 📑 核心参数速查

物理材质主要控制碰撞时的摩擦和弹跳，其关键参数如下：


| 参数             | 作用                                                                                           | 常见值范围                          | 手感参考                                                                                  |
| ---------------- | ---------------------------------------------------------------------------------------------- | ----------------------------------- | ----------------------------------------------------------------------------------------- |
| **动/静摩擦力**  | 动摩擦力：物体滑行时的阻力，数值越大，减速越快。静摩擦力：物体从静止到开始滑动所需克服的力。   | 0\~ 1                               | 0 → 冰面（极其滑溜）<br/>0.4 → 普通地面<br/>1 → 橡胶（阻力巨大）                       |
| **弹性**         | 控制碰撞后反弹的程度，即碰撞前后的能量损失。                                                   | 0\~ 1                               | 0 → 完全没弹性（泥巴）<br/>0.5 → 普通弹力球<br/>1 → 无能量损失（理想状态）             |
| **摩擦结合模式** | 双方碰撞器材质摩擦力的组合方式，是优先级从高到低：**Maximum > Multiply > Minimum > Average**。 | Average, Minimum, Maximum, Multiply | **Maximum**: 用最大值；**Multiply**: 相乘；**Minumum**: 用最小值；**Average**: 取平均值。 |
| **弹性结合模式** | 双方碰撞器材质弹性的组合方式，逻辑与摩擦结合模式相同。                                         | Average, Minimum, Maximum, Multiply | 同上                                                                                      |

> Unity 物理系统（PhysX）的性能与稳定性优先于物理精度，因此有时模拟结果可能不如预期。

### 🧰 应用场景示例

通过调整参数，可以创造出真实的物理质感。以下是几个典型的应用场景：


| 场景         | 配置思路                                                                  | 手感描述              |
| ------------ | ------------------------------------------------------------------------- | --------------------- |
| **冰面关卡** | 设置低摩擦力 (`0.1` 或 `0.0`)，角色在其上会打滑，难以控制移动。           | 滑溜溜，停不下来。    |
| **弹跳机关** | 设置高弹性 (`0.8` 或以上)，用于让角色或机关进行远距离跳跃。               | Q弹，碰一下能弹很远。 |
| **粗糙地面** | 设置较高摩擦力 (`0.6`\~`0.8`)，角色在上面移动会很快停下来，有“抓地感”。 | 真实，有阻力感。      |
| **角色本身** | 设置中等摩擦力和低弹性，作为与其他物理材质交互的基础。                    | 标准手感。            |

> 💡 **平台下落问题**：如果角色下落时按住方向键会粘在墙上，可以通过调整墙壁和角色的物理材质摩擦力（让它们更“滑”点）来解决。

### 🧐 踩坑误区

* **与触发器共用失效**：若同一个物体同时有**触发器**（IsTrigger）和**非触发器**碰撞器，直接将物理材质拖到组件上可能会失效。因为系统默认只给排在第一个的碰撞器赋值，若它恰好是触发器，物理材质就不会生效。
* **无物理材质≠无效果**：没有指定物理材质时，碰撞体会使用一组**默认值**（默认摩擦力 `0.6`，弹性 `0`），而不是毫无效果。
* **导致角色动画异常**：设置了摩擦力为 `0` 的角色，松开移动键后会因惯性继续滑动，导致从“跑步”切换到“待机”的动画**延迟1秒才触发**。此时需要在角色停止移动的脚本中直接设置速度为0，以强制停止。
* **兼容性问题**：`AddForce` 等调用有时会意外忽略物理材质，可尝试改用 `AddRelativeForce` 等方式规避。

### 🚀 性能与稳定优化

* **合理设置物理材质数量**：建议对**静态对象**（地形、墙体）**实例化**材质；对**动态对象**（玩家、敌人）**复用**材质，既能控制数量，又便于统一调整。
* **使用材质组合模式**：利用 `Friction Combine` 和 `Bounce Combine` 处理不同材质交互，可无需代码创建丰富的物理效果。
* **启用全局优化（Project Settings → Physics）**：
  * **无意义层禁用碰撞**：在 `Layer Collision Matrix` 中，直接勾选禁用特定层之间的碰撞，例如玩家层与子弹层完全不需要碰撞。
  * **调整求解迭代次数**：可针对性增加单个刚体的 `Solver Iterations` 属性，来提升当前物体的稳定性与准确性，这对单个复杂物体比全局调高更高效。
* **清理无用的物理组件**：确保每个物体的 `Collider` 和 `Rigidbody` 都是必需的，注意 `Is Trigger` 的碰撞体不参与常规物理模拟。

| **Use Motor**                       | 开启**电机**，主动按指定速度和力量驱动关节旋转           | 适合机关（如传送带滚轮）等需主动驱动的对象                                                  |
| **Motor**                           | **电机**目标速度和最大力量                               | -**Target Velocity**: 电机想达到的速度<br/>- **Force**: 电机能达到的最大扭矩                |
| **Use Limits**                      | 开启**角度限制**，限制关节能在多大角度范围内活动         | 开启后，可设置最小/最大角度（限制关节可动范围），可用于控制门开启角度或限制角色四肢过度弯曲 |
| **Min / Max**                       | 关节可转动的角度区间                                     | **Min**: 最小角度限制，负值<br/>**Max**: 最大角度限制，正值                                 |
| **Break Force**                     | 让关节“折断”所需的**线性力**阈值                       | 力超过阈值 → 移除关节，模拟断裂效果                                                        |
| **Break Torque**                    | 让关节“折断”所需的**扭转力**阈值                       | 启用后需注意性能消耗                                                                        |

### 🌀 行为微调三件套：弹簧、电机与角度限制

你可以通过组合使用这三个功能，获得想要的细腻物理效果：

* **开启 `Use Limits` 限制范围**：为关节设定转动角度的上下限，比如只让门能在0°到90°之间开合。对于角色肢体关节，必须开启“限制”以防止不自然的弯曲。
* **开启 `Use Spring` 实现自动归位**：开启后，物体会自动转向目标角度，模拟“自动归位”效果，比如让一扇推开的门自己缓缓关上。
* **开启 `Use Motor` 主动驱动旋转**：开启后，物体会按指定速度和力量主动旋转，很适合需要自动运行的机关。

### ⚠️ 常见误区与避坑指南

#### 🎡 误区一：旋转轴（Axis）设置错误，导致物体乱转、抖动甚至卡死

这是最常见的错误之一，即旋转轴（Axis）的方向设置不当。例如，**如果你想要一个平开门绕Y轴转动，但你设置的是绕X轴旋转，物理模拟就会失效**。

* **错误表现**：关节无法按预期绕指定轴旋转，出现奇怪抖动。
* **解决方案**：在 `Inspector` 面板的 `Axis` 属性区域，直接点击并拖动坐标轴箭头，或输入数值。**特别注意：模型导入和旋转角度会影响局部坐标轴方向。**

#### 🛠️ 误区二：刚体与碰撞体配置不当引发的物理冲突

当关节连接的物体与玩家角色等碰撞时，经常会出现剧烈抖动，甚至整个结构散架。这往往是因为：连接的刚体质量相差太大，导致解算困难；或是一个刚体被玩家顶住，而它的连接点又被其他物体卡住，两者互相矛盾，物理引擎无法解算，于是抖动。

**解决方案：**

* **调整质量比**：尽量让关节连接的刚体质量相近。如果必须相差悬殊，可尝试启用 `Enable Preprocessing` 并适当调节 `Mass Scale` 属性。
* **检查层级**：使用**刚体组件的Sleeping功能**与**Layer Collision Matrix（层碰撞矩阵）**，禁用无关碰撞。
* **调整参数**：对于核心关节，微调 **Solver Iterations** 和 **Solver Velocity Iterations**，可以提升解算精度和稳定性。

#### 🌲 误区三：链条过长或受到高速冲击时，关节链接处出现分离、拉伸或断裂

当铰链连接的链条过长或被高速挥舞时，链接处容易出现分离、拉伸或断裂。这是因为默认的Break Force/Break Torque参数设置不当，或者链条各段的质量与受力关系不匹配。可以调整刚体的阻力（Drag）角阻力（Angular Drag）增加稳定性；同时，为应对物理误差，尝试使用**DistanceJoint2D**作为辅助。

### 🚀 性能优化建议

* **减少关节数量，必要时物理烘焙**：物理运算是性能开销的一部分，应使用 `GameObject` 的 Active/Deactive 或脚本控制关节的休眠来节省性能。
* **限制参与碰撞检测的物体**：善用 `Layer Collision Matrix`，**避免关节上的物体与不需要交互的物体（如场景装饰）发生碰撞**，减少不必要的物理计算。
* **控制迭代次数**：仅在必要时，为关键物体适当提高 `Solver Iterations`，避免全局设置过高导致性能消耗过大。
* **使用层级结构并降低迭代**：合并网格并合理设置父子层级，可以减少物理计算量。对表现效果影响不大的关节，适当调低刚体组件的迭代次数（Solver Iterations）。

## 刚体

### ⚙️ 核心参数功能速查

这是 Rigidbody 组件最核心的参数及其主要功能：


| 参数                                   | 功能                                                                             | 典型值与说明                                                                                   |
| -------------------------------------- | -------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------- |
| **Mass (质量)**                        | 物体的惯性和与其他物体碰撞时的动量。质量大的物体会推动质量小的物体。             | 默认 1.0。建议不同物体的质量比不超过**10:1**，避免物理模拟不稳定。                             |
| **Drag (阻力)**                        | 线性运动的阻力，用于模拟空气阻力或摩擦力。值越大，物体减速越快。                 | 角色移动建议设为**1-5**，避免“滑冰”感。                                                      |
| **Angular Drag (角阻力)**              | 旋转运动的阻力，值越大，旋转停得越快。                                           | 默认 0.05，保持合理范围（如**0.05-1**）。                                                      |
| **Use Gravity (使用重力)**             | 控制物体是否受重力影响。默认启用。                                               | 通常保持启用，除非需要漂浮等特殊效果。                                                         |
| **Is Kinematic (是运动学)**            | 物体**不受物理引擎控制**，但仍能影响其他物理物体。常用于移动平台或可搬运的道具。 | 启用时，需通过脚本修改 Transform 或 Rigidbody.MovePosition 来控制它。                          |
| **Interpolate (插值)**                 | 平滑物理位置在渲染帧之间的显示，防止物体运动时出现抖动或跳跃。                   | 对跟随摄像机的玩家角色很有用。                                                                 |
| **Collision Detection (碰撞检测模式)** | 决定物理引擎如何检测碰撞，是**精度与性能平衡**的关键。                           | 绝大多数物体保持默认的**Discrete**，快速移动的物体需设为 **Continuous / Continuous Dynamic**。 |
| **Constraints (约束)**                 | 锁定物体的位置或旋转轴，防止物理模拟导致其在某些方向上移动或旋转。               | 常用的是冻结旋转轴的 Z 与 X 轴。                                                               |

> **关于质心与惯性张量**：这两个是更高级的属性。`Mass`、`Drag`、`Angular Drag` 是**最常用且对物理行为影响显著**的核心属性。**质心**影响物体的平衡（例如，一个高重心的物体更容易倾倒），而**惯性张量**决定物体旋转的难易程度。手动调整这些能制作出更真实的物理效果，但非必要不建议修改。

### 📜 常见应用场景

* **3D/2D 平台跳跃 (角色控制)**：通过 `Rigidbody.AddForce()` 施加瞬间跳跃力；并为地面、墙壁等环境物体添加碰撞体。
* **爆炸冲击波 (物理反馈)**：计算爆炸范围内的所有物体，根据距离和方向，用 `Rigidbody.AddExplosionForce()` 模拟冲击力。
* **车辆竞速 (载具模拟)**：在四个轮子的位置施加力，根据车轮转向调整力的方向，模拟车辆的前进、后退和转向。
* **机关解谜 (物体交互)**：利用投掷物触碰压力板触发机关，关键在于为机关部分添加碰撞触发器（`Is Trigger`）。

### ⚡ 核心代码 API (C#)

在脚本中控制刚体的核心方法如下，注意物理相关的代码应放在 `FixedUpdate` 函数中。


| 核心方法                              | 关键参数说明                                                                                                                                      | 示例代码                                                                               |
| ------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------- |
| **`AddForce` (施加力)**               | `ForceMode`：四种施力模式<br/>- `Force`：持续力（默认）<br/>- `Impulse`：瞬间冲量<br/>- `Acceleration`：加速度<br/>- `VelocityChange`：直接改速度 | `rb.AddForce(Vector3.forward * 10f, ForceMode.Impulse);`                               |
| **`AddTorque` (施加扭矩)**            | 施加旋转力，用法与`AddForce`类似。                                                                                                                | `rb.AddTorque(Vector3.up * 100f);`                                                     |

| **`Sleep` / `WakeUp` (睡眠/唤醒)**    | 手动控制刚体进入/退出“睡眠”状态，是**重要的性能优化手段**。                                                                                     | `rb.Sleep();` 或 `rb.WakeUp();`                                                        |
| **`isKinematic` (是否运动学)**        | 控制刚体是否受物理模拟影响。                                                                                                                      | `rb.isKinematic = true;`                                                               |

### ⚠️ 常见误区、问题及解决方案

* **❌ 误区1：通过 Transform 直接移动物体**：直接修改 `transform.position` 会绕过物理引擎，导致穿墙、碰撞错误或动画撕裂。**✅ 正确做法**：对动态物体使用 `AddForce`，对运动学物体使用 `MovePosition`。
* **❌ 误区2：计算力与速度时，未乘以 `Time.deltaTime`**：物理施力应在 `FixedUpdate` 中进行，此时已内置了 `Time.fixedDeltaTime` 的时间步长计算，因此**施加力时不需要再额外乘以 `Time.deltaTime`**。
* **❌ 误区3：将 Collider 和 Rigidbody 都设为触发器 (Trigger)**：如果一个物体**同时勾选 `IsTrigger` 并附加了刚体**，它仍会参与物理模拟，可能导致不可预知的运动和穿透。**✅ 最佳实践**：触发器通常只用于检测，若必须移动它，请使用运动学刚体（`isKinematic = true`）。
* **❌ 误区4：父子级碰撞检测失效**：将一个有刚体的物体设为另一个有刚体的物体的子物体时，`OnCollisionEnter` 等方法可能无法正常触发。**✅ 解决方案**：尽量将刚体放在最高层级的物体上，避免复杂的嵌套刚体结构。
* **⚠️ 问题1：物体下落/运动缓慢**：这可能源于模型的比例不当。Unity 物理系统默认 1 单位 = 1 米，如果你的小车模型长 50 个单位，在物理引擎眼中就是一个 50 米长的庞然大物，运动看起来自然会非常缓慢。**解决方案**：在建模或导入时，确保物体的尺寸与现实世界比例相近。
* **⚠️ 问题2：物体无限弹跳**：当一个物体的某些位置轴被冻结（`Freeze Position`），碰撞时产生的力无法被完全吸收，就会造成持续弹跳。**解决方案**：尝试解冻该物体的 `Freeze Position` 设置。
* **⚠️ 问题3：刚体穿透 (Tunneling)**：当一个物体移动速度过快时，可能会在两个物理帧之间直接穿过另一个物体，从而不发生碰撞。**解决方案**：为子弹等快速移动的物体将碰撞检测模式设为 `Continuous` 或 `Continuous Dynamic`。

### 🚀 性能优化与实践

* **启用“刚体休眠”**：当物体的速度低于阈值时，Unity 会自动让它进入“睡眠”状态。其下落的视觉速度会相当缓慢——物理引擎会暂停对它的计算以节省 CPU 性能。
* **选择合适的碰撞检测模式**：默认的 **Discrete** 性能最佳。仅在必要处（如子弹）使用开销更大的**连续碰撞检测模式**。
* **使用 Solver Iterations (求解迭代次数)**：如果发现物理堆叠或连接过于不稳定，可针对性增加该刚体的 `solverIterations` 属性值（每次物理步进的解算次数），来提升当前物体的稳定性与准确性，这对单个复杂物体比全局调高更高效。
* **善用层碰撞矩阵 (Layer Collision Matrix)**：在 `Project Settings > Physics` 中，通过矩阵**禁用不相关层之间的碰撞检测**，能有效减少无效计算。
* **其他全局优化**：在 Project Settings → Physics 中，可以通过调整 `Fixed Timestep` 等全局参数来平衡物理精度和CPU负载。

### 碰撞模式

#### 一、三种模式对比


| 模式                                 | 原理                                                                                                       | 适用场景                                                     | 性能开销             | 能否防止高速穿透                                |
| ------------------------------------ | ---------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------ | -------------------- | ----------------------------------------------- |
| **Discrete**（离散）                 | 每帧只检测物体从**上一帧位置**到**当前帧位置**的终点处是否有碰撞。                                         | 绝大多数普通物体（角色、静态障碍物、低速移动物）。           | **最低**（默认模式） | ❌ 不能                                         |
| **Continuous**（连续）               | 对**射线**和**连续检测物体**，会扫描从上一帧位置到当前帧位置的整个运动路径。                               | 高速移动且需要**被其他连续物体撞击**的物体（如子弹、飞镖）。 | **中等**             | ✅ 能防止被其他连续物体穿透，但对静态碰撞体无效 |
| **Continuous Dynamic**（动态连续）   | 更严格的连续检测，既能扫描自身运动路径，也能**与所有 Continuous 和 Continuous Dynamic 物体**进行扫描碰撞。 | 需要**被高速子弹击中**的物体（如易碎花瓶、玩家角色）。       | **最高**             | ✅ 能防止与任何连续检测物体的穿透               |
| **Continuous Speculative**（推测式） | 基于推测算法，在运动发生前预测潜在的碰撞并提前处理，开销小于完全的扫描检测。                               | 移动端或性能敏感场景下需要防穿透的物体。                     | **中等偏低**         | ✅ 能防止穿透（精度略低于 Continuous）          |

> **注意**：`Continuous Speculative` 是 Unity 2017.3 之后加入的模式，并非所有旧文档都会提到。

#### 二、典型错误与正确配置


| 错误认知                                              | 正确解释                                                                                                                 |
| ----------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------ |
| 把**所有物体**都设为 Continuous 或 Continuous Dynamic | 会严重降低物理性能。只为**必要的高速物体**开启。                                                                         |
| 为静态碰撞体（无 Rigidbody）设置连续检测              | 连续检测只对**刚体**有效，静态碰撞体永远使用离散检测。                                                                   |
| 认为 Continuous 能防止与**静态网格**的穿透            | Continuous**仅与带有连续检测的刚体**进行扫描。要防止与静态碰撞体的穿透，需改用 Continuous Speculative 或增大碰撞体厚度。 |
| 在 2D 物理中混淆                                      | 2D 刚体（Rigidbody2D）的连续检测模式是`Continuous` 和 `ContinuousDynamic`，用法与 3D 类似。                              |

## 灯光

### 🏮 Unity 光源详解 (Light Component)

Unity的`Light`组件是我们工作的核心。下面我们通过一个表格来总览其主要参数，然后再逐一细说。

#### ⚙️ 核心参数一览

下表汇总了`Light`组件的关键参数及其核心作用：


| 参数分类           | 参数名                 | 核心作用                        | 关键提示                         |
| ------------------ | ---------------------- | ------------------------------- | -------------------------------- |
| **基础**           | `Type`                 | 定义光源的形状和发光方式        | 影响最根本的性能和效果           |
|                    | `Color`                | 设置光源颜色                    | 影响场景色调                     |
|                    | `Intensity`            | 控制亮度                        | 默认值：方向光0.5，其他1.0       |
|                    | `Range`                | 控制光照范围                    | 仅点光源和聚光灯适用             |
|                    | `Spot Angle`           | 控制聚光灯光锥角度              | 仅聚光灯适用                     |
| **模式 (Mode)**    | `Mode`                 | 决定是实时计算还是预先烘焙      | **性能的关键开关**               |
| **阴影 (Shadows)** | `Shadow Type`          | 决定是否投射阴影                | 柔和阴影比硬阴影更耗性能         |
|                    | `Strength`             | 控制阴影的暗度                  | 0为无阴影，1为全黑               |
|                    | `Resolution`           | 控制阴影贴图分辨率              | 更高的分辨率性能消耗更大         |
|                    | `Bias` / `Normal Bias` | 修正阴影渲染时的瑕疵            | 过高会导致阴影与物体分离         |
| **高级**           | `Cookie`               | 在灯光前增加图案遮罩            | 模拟纹理投影，如窗户或树叶的影子 |
|                    | `Culling Mask`         | 控制灯光影响哪些层(Layer)的对象 | 通过剔除无关物体来优化性能       |
|                    | `Render Mode`          | 控制Unity如何渲染该光源         | 覆盖自动分类，手动管理性能       |

接下来，我们对其中最关键的**类型**和**模式**进行深入解读。

#### 🔦 四大光源类型 (Type)

首先，你需要根据场景需求选择合适的光源类型，每种都有其独特的用途和性能消耗。


| 类型                     | 简介                                       | 性能与最佳实践                                                                                                 |
| ------------------------ | ------------------------------------------ | -------------------------------------------------------------------------------------------------------------- |
| **方向光 (Directional)** | 模拟太阳，由方向决定，不受位置影响         | **性能最佳**，一个新场景通常只有一个方向光，但注意旋转它会影响天空盒。                                         |
| **点光源 (Point)**       | 从一点向所有方向发光，模拟灯泡、烛火       | **性能中等，但阴影消耗巨大**。因为它投射阴影需要6张阴影贴图，开销远超聚光灯的1张。适合精确控制范围和使用烘焙。 |
| **聚光灯 (Spot)**        | 从一点向锥形区域发光，模拟手电筒、舞台追光 | **性能优于点光源**，其锥形范围便于进行视锥剔除。常用于室内，也是最灵活的补光工具之一。                         |
| **面光源 (Area)**        | 从一个矩形或圆盘面均匀发光                 | **运行时不支持，仅用于烘焙**。其光照柔和，非常适合模拟大面积光源，如发光的窗户或柔光箱。                       |

#### 💡 三大光照模式 (Mode)

选择了类型后，再通过`Mode`来决定是实时计算性能开销，还是预先“烘焙”来换取性能。


| 模式                | 行为                                                                   | 优缺点                                                                                                                        |
| ------------------- | ---------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------- |
| **实时 (Realtime)** | Unity每帧都会计算光源的亮度和阴影。                                    | ✅ 优点：完全动态，光源可移动，颜色强度可随代码改变。<br/>❌ **缺点：性能开销极高**，动态阴影是渲染中最昂贵的特性之一。       |
| **烘焙 (Baked)**    | 在编辑器中预先计算好光照效果，生成光照贴图。运行时使用贴图，不再计算。 | ✅ 优点：性能消耗极低，渲染效果平滑，可表现复杂的间接光照（颜色反弹）。<br/>❌ 缺点：所有效果都是**静态**的，运行时无法改变。 |
| **混合 (Mixed)**    | 将光影拆分为直接光和间接光，提供“部分动态”方案。                     | ✅ 优点：是表现力与性能的折中选择。例如，动态角色可以接收从静态环境光来的光影效果。<br/>❌ 缺点：配置复杂，且仍有性能成本。   |

### 📜 Lighting 窗口：控制全局光与氛围

除了每个光源的单独参数，`Lighting`窗口负责控制场景的全局光照、环境光和阴影系统，是定调画面风格的核心。

* **环境光 (Environment Lighting)**：定义了没有光源直射区域的默认亮度和颜色。`Source`参数让你可以选择将天空盒、纯色或自定义渐变作为场景的环境照明基础。
* **全局光照 (Global Illumination, GI)**：模拟光在物体表面间反弹的逼真效果。`Lighting`窗口的`Realtime Global Illumination`和`Baked Global Illumination`是其两大技术支柱，决定了如何为场景计算和生成丰富的光影层次。
* **其他设置**：包括控制场景雾效的`Fog`，以及控制阴影显示距离和质量的`Shadow Distance`、`Shadow Cascades`等。

### ⚠️ 常见陷阱 (Pitfalls)

1. **滥用点光源阴影**：实时点光源投射阴影的性能消耗是聚光灯的6倍，应谨慎用于运行时。
2. **误解实时渲染**：应使用最合适的（例如，用面光源营造柔和光影），而不是在性能上“挥霍”。
3. **阴影偏差 (Bias) 问题**：阴影断裂或“漏光”，通常是`Bias`值过高，调低即可。
4. **超出阴影距离**：远处的物体阴影突然消失，是`Shadow Distance`设置过小，调大即可。
5. **忽略剔除设置**：灯光应只照射必要物体。通过`Culling Mask`减少不必要的计算。

### 🚀 性能优化实践 (Optimization)

1. **区分 "逐像素" 与 "逐顶点"**：在质量设置中，优先用最昂贵的**逐像素** 光照服务最核心的物体（如角色），次要的用效率更高的**逐顶点**或**球谐函数 (SH)** 光照处理，这是效率分层的关键。
2. **使用渲染模式 (Render Mode)**：为关键光源标记`Important`以强制其进入最高优先级队列，无关紧要的光源标记为`Not Important`以节省性能。
3. **适度使用阴影**：能不投射阴影的光源就不要投射。并且，只对场景中最重要的光源开启最高质量的阴影贴图。
4. **减少实时光源与阴影**：在移动平台，有每个对象的实时光源数量限制（OpenGL ES 2.0 限制为 4 个），应极力避免视觉冲突。

# 摄像机

### 🔧 一、核心属性与参数详解

摄像机的 Inspector 面板中，部分核心属性的作用和优化建议如下：


| 属性                                          | 作用                                                                                                                                                                 | 优化建议                                                                                                                                                    |
| --------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Clear Flags**                               | 决定当前摄像机如何清除屏幕上的现有内容。每个摄像机在渲染时都会先清空屏幕（如填充纯色、天空盒等），以绘制自己的画面。<br />个人理解：就是摄像机背景，不会出现在场景里 | 当使用多个摄像机叠加UI时，可以设置为`Depth only` 清除深度但不覆盖背景，以保证UI渲染在正确的前景。                                                           |
| **Projection** (`Perspective`/`Orthographic`) | 透视模式模拟真实人眼，有近大远小的效果；正交模式则无视距离，常用于 2D 游戏或**小地图**。                                                                             | 正交模式下的雾效渲染可能不符合预期，因为它会均匀应用在屏幕上。2D 游戏建议使用正交模式。                                                                     |
| **Field of View / Size**                      | **视场角（FOV）** 决定了透视摄像机的视野广度，数值越大，能看到的区域越宽，但边缘的畸变也越严重；**大小（Size）** 则决定了正交摄像机的视野高度。                      | FOV 数值过大会导致画面畸变，过小则视野狭窄。通常设置`60-90` 是标准的舒适范围。                                                                              |
| **Clipping Planes**                           | 定义了摄像机开始和停止渲染的距离范围。超出`Far` 平面的物体会直接被“裁剪”（Cull），不再渲染。<br />个人理解：远面和近面(靠近摄像机)之间的物体才会被渲染             | **远平面（Far）** 的设置对性能影响巨大，`1000` 通常已足够覆盖大多数3D场景。                                                                                 |
| **Viewport Rect**                             | 定义了摄像机画面在屏幕上的位置和大小。                                                                                                                               | 常用于实现**分屏游戏**：为不同玩家单独配置摄像机，并分别设置视口区域。                                                                                      |
| **Depth**                                     | 数字越大，摄像机绘制时越靠前，画面会覆盖 Depth 值较小的摄像机。可以理解为主页面的优先级，数字越大越优先展示                                                          | 配合`Viewport Rect` 使用，可以非常方便地实现 **小地图** 或 UI 效果。                                                                                        |
| **Rendering Path**                            | 决定 Unity 渲染光照和阴影的方式，对画质和性能影响极大。                                                                                                              | **前向渲染（Forward）** 兼容性好但对光源开销大，适合移动端；**延迟渲染（Deferred）** 支持大量实时光源，但需要硬件支持且抗锯齿处理复杂，适用于高端PC和主机。 |
| **Target Texture**                            | 可选的高级特性。可将摄像机的画面输出到一张纹理上，而非直接显示在屏幕。                                                                                               | 经典应用是制作**监控摄像头、后视镜或游戏内的“照片”功能**。                                                                                                |
| **Culling Mask**                              | 允许你指定摄像机只渲染哪些**层（Layer）** 上的物体。                                                                                                                 | 通过勾选无关的层来忽略渲染（如UI层、特效层），是优化性能的利器。                                                                                            |
| **Occlusion Culling**                         | 一种高级优化功能。它**不渲染那些完全被其他物体遮挡** 的物体，即使它们位于摄像机的视锥体内。<br /><br />个人理解就是不渲染指定图层的物体。                            | 非常适合**室内场景**，可以有效减少复杂环境下的渲染压力。                                                                                                    |

### 二、四种渲染模式详解（Render Mode）

`Render Mode` 决定了视频最终显示在什么位置，是 Video Player 最核心的设置之一。


| 渲染模式              | 工作原理                                              | 适用场景                                                      | 性能     | 注意事项                                     |
| --------------------- | ----------------------------------------------------- | ------------------------------------------------------------- | -------- | -------------------------------------------- |
| **Camera Far Plane**  | 视频渲染到摄像机远平面上，出现在所有场景物体**背后**  | 动态背景、远景天空盒、环境氛围                                | **低**   | 需要指定目标摄像机                           |
| **Camera Near Plane** | 视频渲染到摄像机近平面上，出现在所有场景物体**前方**  | 全屏过场动画、UI 叠加特效、视频广告                           | **低**   | 会遮挡所有场景内容，适合过场                 |
| **Render Texture**    | 视频渲染到 Render Texture，可赋值给任意材质的纹理属性 | UI 界面（RawImage）、3D 物体表面（电视/广告牌）、摄像头监视器 | **中**   | 灵活性最高，需要额外创建 Render Texture 资源 |
| **Material Override** | 直接覆盖游戏对象材质上的指定纹理属性                  | 3D 物体表面直接播放视频（无需 Render Texture）                | **中高** | 物体必须有 Renderer 组件                     |

**选择建议**：背景/过场用 Camera Plane，UI 用 Render Texture + RawImage，3D 表面可用两者均可。在性能敏感场景（如移动端），Camera Plane 模式对 GPU 的压力最小。

### 三、视频格式与导入设置

#### 1. 支持的编解码器与格式


| 平台                           | 推荐编解码器            | 容器格式               | 说明                                  |
| ------------------------------ | ----------------------- | ---------------------- | ------------------------------------- |
| **Windows / macOS / 主流平台** | **H.264**               | `.mp4`, `.mov`, `.m4v` | 硬件加速支持最佳，跨平台兼容性最好    |
| **Linux 编辑器**               | **VP8** + Vorbis 音频   | `.webm`                | Linux 不支持 H.264                    |

Unity 支持的常见视频容器格式包括：`.avi`, `.mov`, `.mp4`, `.mpg`, `.webm`, `.wmv` 等。

| 设置项                 | 功能                                                                       |

| `Bitrate Mode`         | 控制视频码率，码率越高画质越好但文件更大                                   |
| `Spatial Quality`      | 是否在转码时压缩图像尺寸（节省空间但会降低画质）                           |

### 四、播放透明视频（Alpha 通道）

Unity 支持两种透明视频方案：


| 方案             | 原理                                                         | 实现方式                                                                          | 适用场景                            |
| ---------------- | ------------------------------------------------------------ | --------------------------------------------------------------------------------- | ----------------------------------- |
| **全局 Alpha**   | 整个视频统一透明度，Render Mode 必须是 Camera Near/Far Plane | Video Player 组件中调整`Alpha` 值（0\~1）                                         | 简单的淡入淡出效果，如渐显/渐隐过场 |
| **每像素 Alpha** | 视频不同区域有不同透明度                                     | 需使用支持 Alpha 的视频编码：**VP8 (WebM)** 或 **Apple ProRes 4444 (macOS Only)** | UI 特效、角色特效、粒子叠加等       |

**透明视频的常见质量陷阱**：

* H.264 不支持 Alpha 通道，用普通解码器强行解码会导致边缘模糊或锯齿；
* 若视频分辨率与 Render Texture 不匹配，缩放会导致画质损失，建议两者保持一致；
* VP8/VP9 解码比 H.264 消耗更大，移动端尤其明显。

### 五、应用场景速览


| 场景                  | 推荐 Render Mode          | 关键要点                                                     |
| --------------------- | ------------------------- | ------------------------------------------------------------ |
| **游戏开场动画/过场** | Camera Near Plane         | 全屏播放，不受场景物体遮挡，Loop 关闭                        |
| **动态远景背景**      | Camera Far Plane          | 置于摄像机最远处，营造环境氛围                               |
| **电视/电脑屏幕播放** | Render Texture            | 创建 Render Texture 赋给屏幕材质，搭配 RawImage 实现 UI 播放 |
| **可交互广告牌**      | Material Override         | 直接覆盖物体材质纹理，适合建筑屏幕等                         |
| **UI 界面视频**       | Render Texture + RawImage | UI 上需要视频组件时必用                                      |
| **360° 全景视频**    | Material Override         | 分配给 Skybox 或 Sphere 材质                                 |
| **特效粒子视频**      | 每像素 Alpha + VP8        | 透明视频实现粒子特效、魔法效果等                             |

### 六、常见误区与踩坑点

#### ⚠️ 误区 1：画面残留 / 画面闪烁（切换视频时出现上一条视频的最后一帧）

**现象**：切换视频时屏幕闪现上一个视频的最后一帧。原因是切换过程中新视频需要 1-2 帧加载，而旧视频的最后画面仍留在 Render Texture 上。

**解决方案**：可在切换新视频前先调用 `Stop()` 停止旧播放，或用脚本将 Render Texture 清除为纯色。对于资源充足、切换不频繁的场景，新建 Render Texture 也能彻底避免残留。

#### ⚠️ 误区 2：网络视频无法播放

* Android 平台的 URL 需要加上 `"http://"` 前缀；
* iOS 需确保 App Transport Security (ATS) 允许非 HTTPS 连接（或在 info.plist 中配置例外）。

#### ⚠️ 误区 3：移动端卡顿/发热严重

* 视频分辨率过大（如 4K）而设备无法硬件解码；
* VP8/VP9 编码的视频在部分平台上被迫使用软件解码，CPU 负载极高；
* 直播流或远程 URL 频繁缓冲。

#### ⚠️ 误区 4：透明视频边缘发黑/锯齿

* 使用了不支持 Alpha 的视频格式（如普通 MP4），应改用 VP8 (WebM)；
* 视频分辨率与 Render Texture 不一致导致采样失真，建议两者尺寸匹配；
* 渲染顺序（Render Queue）或混合模式设置不当，导致半透区域与背景叠加错误。

#### ⚠️ 误区 5：视频声音缺失

* 视频文件中音频轨道未被正确导入：检查 Video Clip Importer 中 `Import Audio` 是否勾选；
* 场景中没有 `AudioSource` 组件：Video Player 需要依赖一个 AudioSource 才能输出音频，可以在挂载 Video Player 的游戏对象上同时挂载 AudioSource，然后将 Video Player 的 `Audio Output Mode` 设置为 `AudioSource` 并指向它。

#### ⚠️ 误区 6：在 WebGL 平台视频无法播放

* `Video Clip` 方式在 WebGL 上不被支持，必须改用 `URL` 方式，将视频放置在服务器并通过 http:// 地址访问。

---

### 七、性能优化方案

#### 1. 编码与分辨率控制


| 优化手段                               | 效果                                     |
| -------------------------------------- | ---------------------------------------- |
| 使用**H.264** 编码替代 VP8/VP9         | 硬件解码支持好，CPU/GPU 负载更低         |
| 降低视频分辨率和帧率                   | 移动端尤其显著，推荐 ≤1080p，帧率 30fps |
| 启用 Video Clip Importer 的`Transcode` | 自动优化为目标平台的格式                 |
| 控制码率（Bitrate）                    | 码率过高浪费性能，过低影响画质           |

#### 2. 加载与资源管理


| 优化手段                      | 效果                                             |
| ----------------------------- | ------------------------------------------------ |
| **预加载/异步加载**           | 提前加载视频资源，避免播放时触发 I/O 延迟        |
| **使用 StreamingAssets**      | 外部存储大视频，避免嵌入构建包增大体积           |
| **及时释放 VideoPlayer 实例** | 调用`Stop()` 并销毁 Render Texture，防止内存泄漏 |
| **启用视频缓存**              | 网络流媒体可缓存数据减少重复读取                 |

#### 3. 多 Video Player 管理

当场景中需要**同时播放多个视频**时，需要更精细地平衡性能与效果：

* 如果同一时间只需要显示一个视频，应采用 **单实例复用** 策略：只维护一个 Video Player 对象，播放新视频前先 `Stop`、清除资源、更换源后再 `Play`，避免多个解码器同时占用资源；
* 如果**必须同时播放多个视频**，优先将次要视频设为较低分辨率，并尽量使用 `Camera Far/Near Plane` 模式（开销低于 Render Texture），严格控制同时活跃的播放器数量。

#### 4. 移动端进阶优化

移动平台需额外关注：

* **Surface 生命周期陷阱**：当 Activity 被系统回收时（如切换 App），若 VideoPlayer 未显式调用 `Stop()` 并置空 `targetTexture`，将导致 OpenGL 上下文异常与黑屏；
* **硬解降级**：当硬件解码失败时（如特定 SoC 不兼容），自动降级为软件解码（使用 UnityWebRequest + 备用方案）；
* **内存水位监控**：监控 VideoPlayer 纹理内存占用（Android 建议 ≤32MB），超阈值时触发纹理回收。

# 脚本

## 生命周期

Awake：初始化时会调用，**只调用一次**。主要是进行对相见的初始化、单例等。
OnEnable：激活的时候会调用。每次对象变为激活时调用。
Start:首次update前会调用，只调用一次。
FixedUpdate：固定时间间隔调用，默认是0.02s。类似定时器
Update: 每一帧都会调用。常用的游戏逻辑处理。
LateUpdate：每一帧在所有脚本的Update方法调用之后调用。类似钩子函数。如摄像机的跟随等。
OnDisable:附着对象变为非激活状态时调用，主要用于注销、保存等操作。
OnDestroy:附着对象被销毁时调用，主要是进行最终清理。

### 激活和销毁的区别


| 维度             | 激活/禁用 (Active/Inactive)                             | 销毁 (Destroy)                              |
| :--------------- | :------------------------------------------------------ | :------------------------------------------ |
| **对象是否存在** | 对象**仍然存在于场景**中，只是被标记为不可见/不参与逻辑 | 对象**被从场景中彻底移除**，引用变为 `null` |
| **内存占用**     | **仍占用内存**（组件、数据都保留）                      | **释放内存**（除非有其他引用阻止垃圾回收）  |
| **能否恢复**     | **可以**，通过 `SetActive(true)` 立即恢复               | **无法恢复**，需重新实例化或从预制体生成    |
| **触发生命周期** | `OnEnable` / `OnDisable`                                | `OnDestroy`                                 |
| **对子对象影响** | 递归影响所有子对象                                      | 递归销毁所有子对象                          |
| **典型用途**     | 临时隐藏物体（如 UI 面板、子弹对象池）、暂停逻辑        | 永久删除物体（如敌人死亡、场景切换清理）    |

对象仍然存在，只是不可见。
仍占用内存
临时隐藏物体，UI面板、暂停等

销毁时
彻底移出场景
不再占用内存
敌人死亡/场景切换清理

## 执行顺序


| 维度                    | 规则 / 说明                                                                                                                               |
| :---------------------- | :---------------------------------------------------------------------------------------------------------------------------------------- |
| **整体阶段顺序**        | **固定**：`Awake` → `OnEnable` → `Start` → `Update` → `LateUpdate` → …（所有脚本先完成同一阶段，再进入下一阶段）                    |
| **同一阶段内（默认）**  | **不确定**，由 Unity 内部决定，无法预测谁先执行                                                                                           |
| **同一阶段内（可控）**  | 依**执行优先级数值从小到大**（升序）执行，数字越小越先执行                                                                                |
| **同一脚本的不同实例**  | **无法单独控制**顺序，只能控制不同脚本类之间的顺序                                                                                        |
| **设置方法① - 编辑器** | `Edit` → `Project Settings` → `Script Execution Order`，添加脚本并调整序号（数字越小越先）                                              |

| **编辑器与代码优先级**  | **编辑器面板中的设置优先**，会覆盖代码中的 `DefaultExecutionOrder`                                                                        |
| **未设置优先级的脚本**  | 全部在**Default Time** 槽位内以**不确定顺序**执行                                                                                         |
| **可靠设计建议**        | - 将不依赖他人的初始化放`Awake`，依赖他人的放 `Start`<br>- 使用集中式管理器手动控制初始化顺序<br>- 避免过度依赖脚本执行顺序来修正代码设计 |

整体阶段：
分阶段，统一执行。先执行所有脚本的Awake，然后再是OnEnable。逐次顺序执行。

同一阶段：
默认是随机执行。
设置优先级的情况下则是数字小的先执行。

同一脚本的不同实例：无法确定顺序

### 定义

### 运算

加法：A(x1,y2) + B(x2,y2) = (x1+x2,y1+y2)。几何意义上就是两个向量构建的平行四边形的对角线。
减法：A(x1,y2) - B(x2,y2) = (x1-x2,y1-y2)。几何意义上就是A的终点到B的终点的连线(平行四边形短的对角线)
乘法：A(x1,y2) * 2 = (x1 * 2,y1 * 2)

#### 一、点乘的数学定义

**点乘（Dot Product）** 是线性代数中两个向量之间的一种运算，结果是一个**标量**（一个数字）。

* **公式**（分量形式）：
  对于向量 `A = (x1, y1, z1)` 和 `B = (x2, y2, z2)`
  `A · B = x1*x2 + y1*y2 + z1*z2`
* **公式**（几何形式）：
  `A · B = |A| * |B| * cosθ`
  其中 `|A|`、`|B|` 是向量的长度（模），`θ` 是 A 与 B 之间的夹角（0\~π）。

#### 二、几何意义 —— 方向关系的“标尺”

从 `cosθ` 的符号和大小可以直观判断两个向量的方向关系：


| `A·B` 的值 | 对应的几何关系             | 夹角 θ            |
| ----------- | -------------------------- | ------------------ |
| **正数**    | 方向大致相同（指向同一侧） | 0° ≤ θ < 90°   |
| **0**       | 互相**垂直**（正交）       | θ = 90°          |
| **负数**    | 方向大致相反（指向相反侧） | 90° < θ ≤ 180° |
| `           | A                          |                    |
| `-          | A                          |                    |

> 当 A 和 B 都是**单位向量**（长度为1）时，`A·B = cosθ`，直接得到夹角的余弦值。

#### 三、在图形编程中的核心意义

点乘在 3D 图形学、游戏开发（Unity/Unreal）、着色器编程中无处不在，主要用来判断方向、角度和投影。

##### 🔹 1. 光照计算（漫反射光照 - Lambert）

* **公式**：`diffuse = max(0, normal · lightDir)`
* 用法：表面法线 `normal` 点乘 光源方向 `lightDir`。结果 >0 表示表面朝向光源，结果越大光照越强；结果为 0 或负值则无光。
* 这是所有实时渲染的基础。

##### 🔹 2. 判断物体的前后关系（视锥/视线）

* 用法：摄像机前向向量 `cameraForward` 点乘 从摄像机指向物体的向量 `dirToObject`。
  * 结果 >0 → 物体在摄像机**前方**。
  * 结果 <0 → 物体在**后方**（可能被剔除）。
* 类似用途：AI 视野检测（前方扇形范围）。

##### 🔹 3. 计算两个向量之间的夹角

* 用法：`θ = arccos( (A·B) / (|A||B|) )`
* 用途：计算转向角度、子弹命中角度、判断是否朝向目标等。

##### 🔹 4. 计算投影长度（投影）

* 用法：`projLength = A · (B / |B|)` 或直接用单位向量。
* 用途：例如将一个点的速度分解到某个轴的方向上。

##### 🔹 5. 镜面高光（Blinn-Phong 简化）

* 用法：半角向量 `H` 点乘法线 `N`，结果用于控制高光强度。

##### 🔹 6. 判定方向相似度

* 用法：例如 Steering Behavior 中，期望方向点乘当前方向，结果越接近 1 表示越一致。

#### 一、叉乘的数学定义

**叉乘（Cross Product）** 是只在三维空间中定义的向量运算，结果是一个**新的向量**，该向量垂直于原来的两个输入向量。

* **公式**（分量形式）：
  对于向量 `A = (x1, y1, z1)` 和 `B = (x2, y2, z2)`
  **text**

  ```
  A × B = ( y1*z2 - z1*y2 ,
            z1*x2 - x1*z2 ,
            x1*y2 - y1*x2 )
  ```
* **公式**（几何形式）：
  `|A × B| = |A| * |B| * sinθ`
  其中 `θ` 是 A 与 B 之间的夹角（0\~π）。

#### 二、几何意义 —— 生成垂直向量与有向面积

叉乘的结果向量 `C = A × B` 具有以下性质：


| 属性         | 含义                                                                                                                 |
| ------------ | -------------------------------------------------------------------------------------------------------------------- |
| **方向**     | 垂直于`A` 和 `B` 所张成的平面，遵循**右手定则**：<br/>四指从 `A` 弯向 `B`（角度 < 180°），拇指方向即为 `C` 的方向。 |
| **长度**     | `                                                                                                                    |
| **交换律**   | `A × B = - (B × A)`，叉乘反交换。                                                                                  |
| **特殊情况** | 若`A` 与 `B` 平行（`sinθ = 0`），则 `A × B = 0`（零向量）。                                                        |

> 右手定则图示：
>
> * 右手坐标系（Unity 世界坐标）：X右，Y上，Z前。
> * `X × Y = Z`，`Y × Z = X`，`Z × X = Y`。

#### 三、在图形编程中的核心意义

叉乘在 3D 图形学、游戏开发、物理引擎中主要用于**计算方向、法线、旋转轴和力矩**。

##### 🔹 1. 计算垂直向量（法线）

* **用法**：已知一个三角形的两条边向量 `AB` 和 `AC`，通过 `AB × AC`（或 `AC × AB`，取决于朝向）得到该三角形的**法线向量**。
* **应用**：生成网格的法线数据，用于光照计算；计算平面方程。

##### 🔹 2. 构建坐标系（基向量）

* **用法**：已知一个方向（例如前向 `forward`）和另一个参考方向（例如世界向上 `up`），通过叉乘补齐完整的局部坐标系的三个轴。
  * `right = up × forward`（归一化后）
  * `trueUp = forward × right`
* **典型场景**：摄像机跟随、导弹制导、`Transform.LookAt` 的内部实现。

##### 🔹 3. 判断方向性（左/右）

* **用法**：对于 `forward` 和 `toTarget`，计算 `cross = forward × toTarget`，然后检查 `cross.y` 的符号（在水平面上）：
  * `cross.y > 0` → 目标在**右侧**。
  * `cross.y < 0` → 目标在**左侧**。
* **应用**：绕 Y 轴旋转的转向判断、UI 箭头指示器。

##### 🔹 4. 计算力矩与角速度（物理）

* **用法**：力矩 `τ = r × F`，其中 `r` 是力臂向量，`F` 是力向量。叉乘结果的大小等于力矩大小，方向是旋转轴方向。
* **应用**：刚体受力产生旋转的物理模拟。

##### 🔹 5. 计算旋转轴（从一个向量旋转到另一个）

* **用法**：旋转轴 = `A × B`（归一化）。`A` 旋转到 `B` 的最短路径旋转轴垂直于两者平面。
* **应用**：`Quaternion.FromToRotation` 的原理，实现物体转向动画。

##### 🔹 6. 判断点是否在三角形内部（重心坐标辅助）

* 通过边缘叉乘符号一致性判断。

#### 插值

##### 一、插值的数学定义

插值的基本形式：给定起点 A**A** 和终点 B**B**，以及一个插值因子 t**t**（通常 0≤t≤1**0**≤**t**≤**1**），计算中间值 P**P**。

P=f(A,B,t)**P**=**f**(**A**,**B**,**t**)当 t=0**t**=**0** 时 P=A**P**=**A**；t=1**t**=**1** 时 P=B**P**=**B**；t=0.5**t**=**0.5** 时 P**P** 在中间位置。

最常见的插值类型是**线性插值（Linear Interpolation, Lerp）**：

* **标量（浮点数）**：
  Lerp(a,b,t)=a+(b−a)×t=(1−t)⋅a+t⋅b**Lerp**(**a**,**b**,**t**)**=**a**+**(**b**−**a**)**×**t**=**(**1**−**t**)**⋅**a**+**t**⋅**b
* **向量**：对每个分量分别线性插值。
  Lerp(A⃗,B⃗,t)=A⃗+(B⃗−A⃗)×t**Lerp**(**A**,**B**,**t**)**=**A**+**(**B**−**A**)**×**t
* **颜色**（RGBA）：对红、绿、蓝、透明度各通道分别线性插值。

##### 二、图形编程中的常见插值类型


| 类型               | 英文         | 数学特征                | 典型用途                               |
| ------------------ | ------------ | ----------------------- | -------------------------------------- |
| **线性插值**       | Lerp         | 匀速直线运动            | 简单移动、褪色、进度条                 |
| **球面线性插值**   | Slerp        | 沿球面最短弧插值        | 旋转（四元数）、摄像机朝向             |
| **平滑插值**       | SmoothStep   | 曲线先快后慢再快（S形） | 自然淡入淡出、摄像机缓动               |
| **三次贝塞尔插值** | Cubic Bezier | 自定义曲线形状          | 动画曲线、材质编辑器的动画曲线         |
| **指数/平滑阻尼**  | SmoothDamp   | 近似物理弹簧阻尼        | 平滑跟随、摄像机平滑跟随（Unity 内置） |

> **注**：对于四元数（Quaternion）旋转，必须用 **Slerp** 而非 Lerp，否则旋转路径会扭曲。

##### 三、在游戏开发中的核心应用

###### 🔹 1. 物体平滑移动

不用 `transform.position = targetPosition`（瞬移），而用 Lerp 每帧靠近一小段，产生跟随或缓动效果。

**csharp**

```
transform.position = Vector3.Lerp(transform.position, targetPosition, Time.deltaTime * speed);
```

###### 🔹 2. 颜色渐变

**csharp**

```
renderer.material.color = Color.Lerp(startColor, endColor, t);
```

###### 🔹 3. 摄像机平滑跟随

使用 `Vector3.SmoothDamp` 或 `Mathf.SmoothDamp` 实现带惯性效果的跟随，避免突兀。

###### 🔹 4. 旋转插值（四元数）

**csharp**

```
transform.rotation = Quaternion.Slerp(transform.rotation, targetRotation, Time.deltaTime * rotateSpeed);
```

###### 🔹 5. 动画混合

角色动画中，根据速度参数在 Idle 和 Run 动画之间插值权重；或使用 `Mathf.SmoothStep` 控制 UI 面板的显示/隐藏动画。

###### 🔹 6. 纹理采样与抗锯齿

GPU 在采样纹理时，根据 UV 坐标在两个相邻像素颜色之间做双线性插值（2D Lerp），使图像放大后不平滑缺失。
